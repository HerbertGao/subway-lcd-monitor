# experience-adaptation 规范

## 目的
experience-adaptation 能力定义 subway-lcd-monitor 的体验适配要求，涵盖响应式布局（LCD 屏含外框、控制面板、页面容器在多种屏幕尺寸下不溢出，并在极窄屏让步于内容完整性）与基础可访问性（SVG 场景语义化、切换控件的 ARIA 状态、表单标签关联、键盘焦点可见）。

## 需求
### 需求:LCD 屏及其外框必须自适应屏幕宽度

LCD 屏（含其外框装饰）在视口宽度不足设计基准（约 960px 加外框）时，必须整体收缩以适应可用宽度，禁止溢出视口或产生横向滚动；在宽屏下保持设计基准尺寸。LCD 屏整体（根元素，含 header、内容区与 footer）在视口宽度充足时维持 3∶1 的宽高比例；当视口过窄、严格 3∶1 会使内容区高度不足以容纳场景内容时，LCD 屏高度允许超过「宽 ÷ 3」——保证场景内容完整、不被裁剪的优先级高于严格维持 3∶1 比例。外框的内边距/边框必须计入其宽度（`box-sizing: border-box`），不得使整体超出可用宽度。

#### 场景:窄屏下显示
- **当** 视口宽度小于设计基准
- **那么** LCD 屏连同外框整体收缩至适应可用宽度，不溢出、不产生横向滚动条

#### 场景:宽屏下显示
- **当** 视口宽度足够容纳设计基准
- **那么** LCD 屏整体以设计基准尺寸（约 960×320，3∶1）显示

#### 场景:极窄屏下比例让步以容纳内容
- **当** 视口过窄，使严格 3∶1 的内容区高度不足以容纳场景内容
- **那么** LCD 屏高度增加至足以容纳场景内容，宽高比可大于 3∶1，场景内容完整显示、不被裁剪

### 需求:所有 LCD 场景在窄屏下必须不溢出内容区

LCD 屏内的全部场景——含 SVG 的场景与非 SVG 的文本场景——在 LCD 屏等比收缩后，均必须保持在内容区范围内显示，禁止内容溢出内容区或相互挤压重叠。非 SVG 场景的字号、内边距、间距必须随尺寸自适应（如使用 `clamp()`），多元素行必须能换行。

#### 场景:窄屏下查看 SVG 场景
- **当** 在窄视口下查看含 SVG 的场景
- **那么** 该 SVG 借 `viewBox` 等比缩放，完整显示在内容区内、不溢出

#### 场景:窄屏下查看非 SVG 场景
- **当** 在窄视口下查看非 SVG 的文本场景（到站提示）
- **那么** 其文字与内边距随尺寸收缩，完整显示在内容区内、不溢出、不相互挤压重叠

### 需求:控制面板与页面容器必须自适应屏幕宽度

控制面板禁止使用固定宽度，必须随可用宽度自适应并设上限（不超过 960px）；其内部分区在窄屏下必须能正常换行或纵向堆叠。页面容器在窄屏下不得因内边距导致内容被挤压溢出。

#### 场景:窄屏下显示控制面板
- **当** 视口宽度小于控制面板上限宽度
- **那么** 控制面板宽度收缩至适应可用宽度，内部分区正常换行/堆叠，不溢出

### 需求:SVG 场景必须对读屏软件可理解

含 `<svg>` 的场景组件必须为该 `<svg>` 提供无障碍语义：`<svg>` 须带 `role="img"`，并通过 `<title>` 提供简短名称、`<desc>` 提供描述当前线路与运行状态的文本，二者经 `aria-labelledby` 关联到该 `<svg>`。

#### 场景:读屏软件访问 SVG 场景
- **当** 读屏软件浏览/访问到某个含 SVG 的场景
- **那么** 它把该 SVG 作为单一图像呈现，并念出标题与包含当前线路、站点、方向的描述

### 需求:自动/手动切换控件必须暴露开关状态

控制列车自动/手动模式的切换控件必须以开关语义对读屏软件暴露状态——使用 `role="switch"` 与随实际模式变化的 `aria-checked`，并配一个不随可见文本变化的稳定可访问名。

#### 场景:读屏软件读取自动/手动控件
- **当** 读屏软件访问处于自动模式的切换控件
- **那么** 它以稳定名称播报该控件，并播报其开关处于「开」；处于手动模式时播报「关」

### 需求:画面切换按钮组必须暴露分组与选中状态

画面切换的一组按钮必须以 `role="group"` 加组标签包裹；组内每个按钮必须以 `aria-pressed` 表达自身是否为当前选中的画面。

#### 场景:读屏软件读取画面切换按钮组
- **当** 读屏软件访问画面切换按钮组
- **那么** 它播报该组的标签，并对每个按钮播报其是否为当前选中（`aria-pressed`）

### 需求:表单控件必须与其标签关联

下拉选择控件必须与其可见标签正确关联——每个控件有唯一 `id`，对应 `<label>` 通过 `for` 指向该 `id`。

#### 场景:读屏软件读取下拉控件
- **当** 读屏软件访问城市/线路/方向的下拉选择控件
- **那么** 它念出该控件对应的可见标签文本

### 需求:键盘焦点必须可见

所有可交互控件（按钮、下拉框）在通过键盘获得焦点时，必须显示清晰可见的焦点指示；禁止在未提供等效可见焦点样式的情况下移除浏览器默认焦点轮廓。

#### 场景:键盘 Tab 浏览控件
- **当** 用户用键盘 Tab 键在可交互控件间移动焦点
- **那么** 当前获得焦点的控件显示清晰可见的焦点指示
