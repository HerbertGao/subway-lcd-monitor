## 新增需求

### 需求：主题数据结构

系统必须定义 Theme（主题）数据结构，包含两大部分：

**视觉样式 (visual)**：
- `colors`: 颜色配置（background、foreground、lineColor 等）
- `fonts`: 字体配置（stationName、info 等字体族和大小）
- `stationMarker`: 站点标记形状（`"circle"` | `"square"` | `"diamond"`）
- `lineCap`: 连线端点样式（`"round"` | `"square"`）

**行为配置 (scenes)**：
- 每个列车运行状态对应的场景列表及时长配置
- 场景切换的过渡动画类型（如 `"fade"` | `"slide"`）

#### 场景：完整主题定义

- **当** 加载一个包含所有字段的主题配置
- **那么** 系统必须正确解析视觉样式和行为配置的所有字段

#### 场景：部分主题定义

- **当** 主题配置中某些字段缺失
- **那么** 系统必须使用默认主题的对应值进行填充

### 需求：主题解析优先级

系统必须按以下优先级解析主题：线路级主题 > 城市级主题 > 全局默认主题。

解析过程为逐层合并：先加载默认主题作为基础，再用城市默认主题覆盖，最后用线路绑定主题覆盖。

#### 场景：线路指定主题

- **当** 线路数据中指定了 themeId
- **那么** 系统必须使用该主题的配置，缺失字段回退到城市默认主题，再回退到全局默认主题

#### 场景：仅城市默认主题

- **当** 线路数据中未指定 themeId，但城市 network 中指定了 defaultThemeId
- **那么** 系统必须使用城市默认主题，缺失字段回退到全局默认主题

#### 场景：无任何主题指定

- **当** 线路和城市均未指定主题
- **那么** 系统必须完全使用全局默认主题

### 需求：CSS 变量注入

主题的视觉样式必须通过 CSS 自定义属性（CSS Variables）注入到 LCD 容器中，场景组件通过引用这些变量实现样式适配。

#### 场景：主题切换

- **当** 用户切换到使用不同主题的线路
- **那么** LCD 容器的 CSS 变量必须立即更新为新主题的值，所有场景组件的样式必须随之变化

### 需求：场景组件覆盖

主题必须能够为特定场景提供自定义组件，覆盖默认的场景组件。未覆盖的场景必须继续使用默认组件。

#### 场景：部分覆盖

- **当** 主题为 ARRIVING 状态的到站场景指定了自定义组件，但未为 RUNNING 状态的场景指定
- **那么** ARRIVING 状态必须使用主题的自定义组件，RUNNING 状态必须使用默认场景组件

#### 场景：无覆盖

- **当** 主题未指定任何自定义场景组件
- **那么** 所有场景必须使用默认组件，仅通过 CSS 变量改变视觉样式

### 需求：默认主题

系统必须内置一个全局默认主题，提供完整的视觉样式和行为配置，确保在无任何自定义主题时系统仍能正常运行。

#### 场景：零配置启动

- **当** 数据文件中未指定任何主题配置
- **那么** 系统必须使用默认主题渲染 LCD 显示屏，所有场景正常显示
